# Rulebook 模組化提案 —— 提升 AI 遵規精準度 × 降低 system_prompt 負擔

> 本文件為設計提案,**尚未實作**。
>
> **核心目的**:讓每次 AI 呼叫只收到與該任務直接相關的規則,消除雜訊干擾、提升遵規精準度。
>
> 精準度問題的根源是**信噪比**:裁判擲骰呼叫①真正需要的規則約佔整本 15–20%,其餘 80–85% 對該任務毫無意義——模型的注意力被稀釋到相關規則上的比例僅剩 1/5。記憶萃取呼叫③更極端:相關規則佔比為 **0%**(格式已 inline 在程式碼),整本 rulebook 在此呼叫中純屬雜訊。此外,依據「Lost in the Middle」效應,長 context 下模型對中段內容的遵循率會顯著下滑——把不需要的規則一起送入,等於讓關鍵規則被埋在雜訊中間失焦。
>
> 省 token 是副產品;**讓模型只讀它該讀的,才是精準度的根本解方**。

## 1. 診斷:精準度瓶頸在「每次 API 呼叫都整本送出」

- 真正開銷在 `gm_bot.py:260-264` 的 `safe_generate_content`:**每次呼叫**都把全域 `system_prompt`(整本 rulebook)塞進 `system_instruction`。
- 而且每處理一則玩家訊息,最多會呼叫 **3 次** AI,各自需求差異極大:

| AI 呼叫 | 程式位置 | temperature | 實際需要的規則 | 目前拿到 |
|---|---|---|---|---|
| ① 裁判擲骰 | `gm_bot.py:451` | 0.1 | 只要第三部分(D100/調整值/優劣勢) | 整本 |
| ② 說書人敘事 | `gm_bot.py:521` | 0.7 | 依狀態不同(創建 vs 遊玩) | 整本 |
| ③ 記憶萃取 | `gm_bot.py:595` | 0.2 | 幾乎不需要(格式已 inline 在 `gm_bot.py:582-590`) | 整本 |

①和③現在是把整本規則白送,浪費最大。

## 2. 提案:把 rulebook 拆成 4 個模組,依呼叫類型組裝

### 模組切分(原文搬移,不改字)

**`common.md` —— 說書人通用(創建+遊玩都載入)**

| 來源段落(現 rulebook.md) | 理由 |
|---|---|
| 開頭句「生成一個無限流模擬器…」 | 框架定位 |
| `## 1. 世界背景`(10 條) | 副本分級/結局/道具/休息時間等基礎世界規則,兩階段都引用 |
| `## 3. NPC 面板` | 創建生成初始 NPC、遊玩遇新 NPC 都要用 |
| `## 5. 其他設定`(10 條) | 回合制、輸出排版、emoji/markdown、禁 HTML、文末選項規則 |
| 文末選項規則(blockquote) | 每則敘事都需要 |
| `第一部分 1. 你的身分` | GM 身分與雙視角 |
| `第一部分 4. 視角隔離鐵律` + `4-1 資訊解鎖鐵律` | secret 防暴雷,創建與遊玩都必須在 |
| `第二部分 對話與回合制推進協定` | 靜默/標記統整,通用 |
| `第四部分 角色卡與世界觀資料庫更新協定` | **JSON 存檔標籤協定,存檔命脈,必須在** |

**`creation.md` —— 僅「等待玩家資料 / 副本生成中」**

| 來源段落 | 理由 |
|---|---|
| `## 2. 初始設定` | 開場劇情/主角面板/第一次選中 |
| `第零部分 玩家偏好與副本生成限制`(含題材清單、黑名單、優先級) | 只在生成副本時需要,**這串最長,遊玩時甩掉省最多** |
| `第一部分 2. 遊戲狀態判定` | 前置引導/開始跑團流程 |
| `第一部分 3. 副本建立規則`(public/secret 結構) | 建立副本 |
| `第一部分 5. 開場輸出格式鐵律` | 開場輸出 |
| `第一部分 6. Game_World 存檔規則`(public/secret 同時生成) | 建立副本 |
| `第一部分 7. 怪物生成規則`(5 隻) | 建立副本 |

**`gameplay.md` —— 僅「副本進行中」及之後**

| 來源段落 | 理由 |
|---|---|
| `第一部分 8. 正式跑團期`(禁止重生成副本) | 遊玩鎖定 |
| `第五部分 視角隔離與物品歸屬`(防通靈/隔空取物) | 多人遊玩 |
| `第六部分 角色成長`(積分/購買時間) | 通關後 |
| `第七部分 主動檢索與連鎖反應` | 遊玩判定 |
| `第八部分 怪物遭遇與調查揭露鐵律` | 遊玩遭遇 |

**`dice.md` —— 裁判呼叫 + 遊玩說書人**

| 來源段落 | 理由 |
|---|---|
| `第三部分 數值隨機性與難度判定`(D100/調整值/優劣勢/dice_request) | 裁判① 判難度;遊玩② 敘述「慘痛代價」尺度 |

### 每個 AI 呼叫載入什麼(`system_instruction`)

| AI 呼叫 | 位置 | 組裝 | 相對現在 |
|---|---|---|---|
| ① 裁判擲骰 | `gm_bot.py:451` | `dice.md` | 整本 → 只剩第三部分 |
| ② 說書人(創建) | `gm_bot.py:521` | `common` + `creation` | 甩掉遊玩各部 + dice |
| ② 說書人(遊玩) | `gm_bot.py:521` | `common` + `gameplay` + `dice` | 甩掉第零部分長清單 + 創建結構 |
| ③ 記憶萃取 | `gm_bot.py:595` | **空字串(無)** | 格式已 inline,整本 → 0 |

**預估節省**:①③ 的 `system_instruction` 砍約 80%;遊玩時的 ② 砍約 30–40%(主要甩掉第零部分那串很長的題材清單)。

**精準度角度**:裁判呼叫① 的信噪比從約 1:5(整本)提升到接近 1:1(只送 `dice.md`),是最直接的遵規精準度量化指標。記憶萃取呼叫③ 從信噪比 0:100% 改為空字串,雜訊歸零。

## 3. 程式改動範圍(小、風險可控)

1. **`safe_generate_content` 加參數**:把寫死的全域 `system_prompt` 改成
   `async def safe_generate_content(prompt_text, system_instruction, temperature=0.7)`,
   呼叫端傳入組裝好的字串。共 3 個呼叫點要改。
2. **啟動時載入模組到 dict**:`RULES = {name: open(...).read() for name in [...]}`,一樣只讀一次。
3. **加組裝函式**:`build_system_instruction(call_type, current_status)` 回傳 `"\n\n".join(...)` 對應模組。
4. regex 抽取 JSON 的邏輯與 system prompt 無關,不受影響。

## 4. 要特別保留、別弄壞的東西

- **JSON 存檔標籤協定(第四部分)必須進 `common`**:創建和遊玩的說書人都會輸出 ` ```json game_world ` / `characters` 等標籤,少了它存檔會壞。
- **視角隔離鐵律(第一部分 4)放 `common`**:只要有 secret 出現的場合(創建+遊玩)都要在。
- **擲骰「必須依結果敘事」鐵則** 目前 inline 在 `gm_bot.py:476-484`,不依賴 rulebook,安全;但「失敗付出慘痛代價」的尺度在第三部分,所以遊玩 ② 要帶 `dice`。

## 5. 替代/互補方案

若不想拆檔,Gemini 也支援 **context caching**(把固定 `system_instruction` 快取,後續呼叫付較低的快取讀取價)。但 Flash 有最低 token 門檻與 TTL 成本,且本需求本來就是「不同呼叫要送不同內容」,模組化更貼合。兩者可並用:模組化先砍量,再對高頻 `common` 做快取。
